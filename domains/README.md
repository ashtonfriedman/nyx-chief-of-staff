# Domains

Recurring areas of your work — people, teams, products, services, repositories.

## What Goes Here

| Subfolder | Contents | Example |
|-----------|----------|---------|
| `people/` | Person notes — working style, role, communication preferences | `people/sarah/sarah.md` |
| `products/` | Product/service notes — ownership, lifecycle, dependencies | `products/core-api/core-api.md` |
| `stakeholders/` | Stakeholder mapping — roles, influence, communication cadence | `stakeholders/stakeholders.md` |
| `services/` | Service architecture — endpoints, dependencies, data flow, deployment | `services/auth-service.md` |
| `repos/` | Repository notes — structure, build commands, test commands | `repos/main-repo.md` |

## Conventions

- One note per entity. If "Sarah" already has a note, update it.
- Use wikilinks (`[[note-name]]`) to cross-reference between domains, initiatives, and expertise.
- Person notes include what matters for working with them: communication style, areas of ownership, preferences.
- Keep notes current. Stale context is worse than no context.
